## Executive Summary

WAL (Write-Ahead Logging) mode provides **significant performance improvements** over the default DELETE journal mode for the control-plane database:

| Metric | DELETE Mode | WAL Mode | Improvement |
|--------|-------------|----------|-------------|
| Single writer throughput | 195 writes/s | 324 writes/s | +66% |
| Concurrent writes (8 threads) | 178 writes/s | 340 writes/s | **+91%** |
| Mixed workload writes | 190 writes/s | 288 writes/s | **+52%** |
| Mixed workload reads | 2.8 reads/s | 823 reads/s | **+29,307%** |
| Lock wait p95 | 7.1ms | 3.4ms | **-53%** |

## Key Findings

### 1. Lock Contention (Primary Goal)
- **p95 lock wait reduced from 7.1ms to 3.4ms** (-52.68%)
- Mean lock wait improved from 25.1ms to 14.8ms
- Maximum lock wait spikes reduced from 3.97s to 2.24s

### 2. Transaction Throughput
- Concurrent write throughput nearly doubled (+91.15%)
- Single-writer performance improved 66%
- Most dramatic improvement in read concurrency during writes

### 3. Read/Write Contention
- DELETE mode: readers heavily blocked by writers (2.8 reads/s)
- WAL mode: readers proceed concurrently (823 reads/s)
- This is the most dramatic improvement—WAL allows concurrent reads during writes

## Edge Cases & Caveats

### Checkpoint Behavior
- WAL files accumulate until checkpointed
- Checkpoint operation took <0.1ms in testing (no active writers)
- Under load, checkpoint may briefly block writes
- **Recommendation:** Enable auto-checkpoint or schedule periodic `PRAGMA wal_checkpoint(TRUNCATE)`

### Disk Usage
- WAL mode adds `-wal` and `-shm` files alongside main database
- `-shm` file: 32KB (shared memory for coordination)
- `-wal` file: grows unbounded without checkpointing
- **Mitigation:** Auto-checkpoint triggers at 1000 pages by default

### Not Suitable For
- Network filesystems (NFS, SMB)—WAL requires shared memory
- Read-only filesystem access
- Databases that must be a single file

